PHASE 3 — MACHINE LEARNING SIGNAL LAYER
----------------------------------------------------------------
Goal: Add ML models that generate smarter signals than rule-based
strategies alone. ML does not replace strategies — it scores them.

Why this order (XGBoost first, then LSTM):
- XGBoost works well on small tabular datasets
- Faster to train and easier to debug than deep learning
- Gives you a working ML signal before adding complexity
- LSTM adds sequential pattern recognition on top

Model 1: XGBoost Signal Classifier
What it does:
- Takes technical indicator features as input
- Predicts probability that price goes up in next N days
- Outputs a score from 0 to 1 (converted to -1 to +1)

Features (inputs to the model):
- RSI value
- MACD histogram value
- Bollinger Band position (0 = lower band, 1 = upper band)
- Volume ratio vs 20-day average
- 1-day, 5-day, 20-day rolling returns
- 20-day rolling volatility
- IMPORTANT: All features must use only past data, never future
  This is called lookahead bias — the most common mistake in
  student quant projects. It makes backtests look great but
  the model is useless in production.

Training process:
- Label data: 1 if price goes up next 5 days, 0 if it goes down
- Split: 70% train, 15% validation, 15% test
- Evaluate with accuracy, precision, recall, and Sharpe ratio
  of trades generated by the signal

Model 2: LSTM Price Sequence Predictor
What it does:
- Takes a sequence of N past days of price data
- Predicts the expected return over the next N days
- Better than XGBoost at capturing sequential patterns

Why LSTM fits this project:
- Price data is a time sequence — LSTM is built for sequences
- Similar to CNN architecture you already know in PyTorch
- Training loop is identical to what you built for the CNN
- Roughly 20% harder than CNN to implement

Architecture:
- Input: sequence of 60 days of OHLCV + indicator features
- LSTM layers: 2 stacked LSTM layers, hidden size 128
- Output layer: single value (predicted N-day return)
- Loss function: MSE for regression
- Framework: PyTorch (same as your CNN project)

Model 3: Regime Classifier
What it does:
- Classifies what type of market the ticker is currently in
- Switches which strategy is active based on regime

Regimes:
- Trending market → activate Momentum strategy
- Sideways/ranging market → activate Mean Reversion strategy
- High volatility → reduce position size
- Low volatility → normal position sizing

How to build it:
- Features: rolling volatility, autocorrelation of returns,
  trend strength (ADX indicator)
- Simple classifier: logistic regression or random forest
- Label training data manually based on known market periods

Signal combination at end of Phase 3:
- XGBoost score + LSTM prediction + regime adjustment
- Combined into single Price Signal Score: -1.0 to +1.0
- This score goes to Signal Combiner (Phase 4)

New FastAPI endpoints:
- GET /ml-signal?ticker=AAPL — returns XGBoost + LSTM combined score
- GET /regime?ticker=AAPL — returns current market regime
- POST /train?ticker=AAPL — retrain models on latest data

Libraries:
- scikit-learn — XGBoost, regime classifier, evaluation metrics
- xgboost — gradient boosting classifier
- torch — LSTM model (PyTorch, same as CNN project)
- joblib — save and load trained models

Timeline: 3-4 weeks

PHASE 4 — NLP AND SENTIMENT LAYER
----------------------------------------------------------------
Goal: Build the second independent signal dimension. Sentiment
analyzes how the market feels about a stock. Completely separate
from price — a second opinion.

Why sentiment matters:
- Price data alone is what everyone uses
- Alternative data like sentiment is where real alpha is
- CEO language in earnings calls predicts moves before price moves
- This is what firms like Citadel actually do with NLP

Model: FinBERT
- Pretrained financial sentiment model on HuggingFace
- Trained specifically on financial text (not general text)
- Input: any financial sentence or headline
- Output: POSITIVE / NEGATIVE / NEUTRAL + confidence score
- Load with: from transformers import pipeline
- 3 lines of code to get a sentiment score from any text
- No training required — plug and play inference only

Data sources for sentiment:

1. Financial News Headlines
   - Source: NewsAPI (free tier, 100 requests/day)
   - Fetch last 24-72 hours of headlines for the ticker
   - Score each headline with FinBERT
   - Aggregate into single news sentiment score

2. Earnings Call Transcripts
   - Source: scrape from Seeking Alpha or Motley Fool
   - Most recent quarter transcript
   - Score each sentence in the transcript
   - Aggregate into earnings sentiment score
   - Calculate sentiment delta vs previous quarter
   - Delta is the most powerful signal — tone shift predicts moves

3. Reddit Social Sentiment
   - Source: Reddit PRAW API (free)
   - Subreddits: wallstreetbets, investing, stocks
   - Detect unusual mention volume spikes for a ticker
   - Score post and comment sentiment with FinBERT

4. Fed Statement Analysis (LLM-based)
   - Source: Federal Reserve website (public)
   - Use an LLM to classify language as hawkish or dovish
   - Hawkish = rate hikes likely = negative for growth stocks
   - Dovish = rate cuts likely = positive for growth stocks

Sentiment Delta (most important signal):
- Compare current earnings tone vs last quarter tone
- If tone shifts from +0.4 to -0.2, delta = -0.6
- Strong negative delta = short signal even if price looks fine
- This is exactly what institutional NLP teams at hedge funds do

Output: single Sentiment Signal Score from -1.0 to +1.0

New FastAPI endpoints:
- GET /sentiment?ticker=AAPL — returns combined sentiment score
- GET /news-sentiment?ticker=AAPL — news headlines breakdown
- GET /earnings-sentiment?ticker=AAPL — transcript analysis
- GET /sentiment-delta?ticker=AAPL — QoQ sentiment shift

Libraries:
- transformers — HuggingFace FinBERT model
- torch — model inference backend
- newsapi-python — news headline fetching
- praw — Reddit API
- requests + beautifulsoup4 — earnings transcript scraping

PHASE 5 — SIGNAL COMBINER
----------------------------------------------------------------
Goal: Merge the price signal and sentiment signal into one final
trading decision. This is the core intelligence of the platform.

How it works:
- Price Signal Score:     -1.0 to +1.0  (weight: 60%)
- Sentiment Signal Score: -1.0 to +1.0  (weight: 40%)
- Combined Score = (price x 0.6) + (sentiment x 0.4)

Decision thresholds:
- Score above +0.3  → BUY signal
- Score below -0.3  → SELL signal
- Score between -0.3 and +0.3 → HOLD (conflicting, stay out)

The HOLD signal is as valuable as BUY or SELL.
When price and sentiment conflict, staying out avoids bad trades.
This is the key insight that separates this from simple strategies.

Position sizing via Kelly Criterion:
- Kelly formula: f = (bp - q) / b
  where b = odds, p = win probability, q = loss probability
- Uses ML model confidence as win probability input
- Higher confidence signal = larger position size
- Prevents betting too much on uncertain signals

Weights are tunable:
- Default 60/40 but can be adjusted per ticker or market regime
- In high-volatility regimes, sentiment weight increases
- In trending markets, price signal weight increases

New FastAPI endpoint:
- GET /combined-signal?ticker=AAPL — returns final decision,
  position size, and breakdown of contributing scores

Timeline: 1 week (depends on Phases 3 and 4 being complete)


----------------------------------------------------------------
PHASE 6 — RISK MANAGEMENT MODULE
----------------------------------------------------------------
Goal: Protect the portfolio. No strategy works 100% of the time.
Risk management is what keeps losses from being catastrophic.

Components:

1. Position Sizing (Kelly Criterion — already in Phase 5)

2. Max Drawdown Limit
   - Set a maximum portfolio loss threshold (e.g. 15%)
   - If portfolio drops more than this, stop all trading
   - Prevents a bad strategy from blowing up the whole portfolio

3. Value at Risk (VaR)
   - Statistical measure of worst expected loss
   - "With 95% confidence, we will not lose more than X% in 1 day"
   - Calculated from historical return distribution
   - Standard measure used by every institutional risk desk

4. Portfolio Exposure Limits
   - Max % of portfolio in a single stock (e.g. 20%)
   - Max % in a single sector (e.g. 40%)
   - Prevents concentration risk

5. Slippage Modeling
   - Real trades do not fill at exact signal prices
   - Model realistic fill prices based on bid/ask spread
   - Currently missing from the Java engine — needs to be added

6. Transaction Cost Modeling
   - Commissions reduce realized returns
   - Model realistic costs per trade
   - Strategies that look profitable before costs may not be after

Timeline: 2 weeks

PHASE 7 — PORTFOLIO OPTIMIZER
----------------------------------------------------------------
Goal: When running multiple strategies or stocks simultaneously,
find the optimal allocation that maximizes risk-adjusted return.

Components:

1. Markowitz Mean-Variance Optimization
   - Classic portfolio theory from 1952, still used everywhere
   - Input: expected returns + covariance matrix of assets
   - Output: optimal weight for each position
   - Minimizes portfolio variance for a given return target
   - Uses numpy linear algebra (matrix operations)

2. Sharpe Ratio Maximization
   - Sharpe = (return - risk free rate) / standard deviation
   - Optimize weights to maximize this ratio
   - Always optimize for risk-adjusted return, not raw return
   - This is how quant funds evaluate strategies

3. Efficient Frontier
   - Plot all possible portfolios in return/risk space
   - Efficient frontier = portfolios with best return per unit risk
   - Visualize in frontend as scatter plot

4. Monte Carlo Simulation
   - Run the backtest 10,000 times with randomized parameters
   - See distribution of outcomes, not just one result
   - Answers: what is the range of realistic outcomes?
   - Prevents over-relying on one lucky backtest result

5. Multi-Strategy Comparison
   - Run momentum, mean reversion, and factor strategies together
   - Compare Sharpe, drawdown, win rate, cumulative return
   - Show side-by-side in frontend dashboard

6. Correlation Matrix
   - Show how holdings move together
   - Low correlation between positions = better diversification
   - Visualize as heatmap in frontend

Libraries:
- scipy.optimize — portfolio optimization
- numpy — matrix operations for covariance
- pandas — return calculations

Timeline: 2-3 weeks

PHASE 10 — AGENTIC AI LAYER
----------------------------------------------------------------
Goal: An autonomous agent that orchestrates all the components
built in Phases 1-9 to conduct quant research automatically.
This is the most advanced feature — only build after everything
else is complete and working.

What the agent does:
- User asks a natural language question
- Agent breaks it into subtasks
- Agent calls the right tools in the right order
- Agent interprets results and decides next steps
- Agent reports findings in plain English

Example interaction:
  User: "Find the best strategy for NVDA right now"

  Agent:
  Step 1 → Fetch NVDA data via yfinance (data tool)
  Step 2 → Detect market regime (regime classifier tool)
  Step 3 → Run all available strategies (backtester tool)
  Step 4 → Pull FinBERT sentiment on last 48hr news (sentiment tool)
  Step 5 → Combine signals (signal combiner tool)
  Step 6 → Run walk-forward validation (validation tool)
  Step 7 → Compare Sharpe ratios across strategies (portfolio tool)
  Step 8 → Report: "Momentum strategy shows Sharpe 1.8,
            sentiment is positive +0.6, recommend LONG
            with 8% position size based on Kelly Criterion"

Why this is impressive:
- Agent is orchestrating real quant research tools, not just chatting
- Every tool it calls was built by you and does real work
- The combination of autonomous reasoning + real financial models
  is genuinely novel for a student project
- Directly mirrors what quant research teams at hedge funds do

Architecture:
- Use Claude API or OpenAI API as the reasoning engine
- Define each module (data fetch, signal, sentiment, backtest,
  portfolio) as a tool the agent can call
- Agent decides which tools to call and in what order
- Tool results fed back to agent for interpretation
- Agent generates final research report

Tools the agent will have:
- fetch_data(ticker, period) → OHLCV dataframe
- run_strategy(ticker, strategy_name) → backtest results
- get_price_signal(ticker) → ML signal score
- get_sentiment(ticker) → FinBERT sentiment score
- get_combined_signal(ticker) → final combined decision
- optimize_portfolio(tickers) → optimal weights
- run_monte_carlo(ticker, strategy) → outcome distribution
- get_regime(ticker) → current market regime

Option 2 — Strategy Generator Agent (also valuable):
- User describes strategy in plain English
- "Buy when RSI is oversold and news sentiment turns positive"
- Agent translates description into strategy parameters
- Automatically runs backtest and reports results
- No coding required from the user

Option 3 — Autonomous Research Loop (most advanced):
- Agent picks tickers to analyze
- Runs strategies, evaluates results
- Decides what to test next based on findings
- Iterates until it finds a statistically significant edge
- Reports full research findings with evidence

Libraries:
- anthropic or openai — LLM reasoning engine
- langchain or custom tool loop — agent orchestration
- All existing FastAPI endpoints become agent tools

Timeline: 4-6 weeks (after all previous phases complete)

----------------------------------------------------------------
CRITICAL RULES — READ BEFORE EVERY SESSION
----------------------------------------------------------------

1. NO LOOKAHEAD BIAS
   Every feature and indicator must use only past data.
   Never let the model see future prices during training.
   This is the most common mistake in student quant projects.
   It makes backtests look great but the model fails in production.

2. WALK-FORWARD VALIDATION IS MANDATORY
   Never evaluate a strategy on the same data it was trained on.
   Always test on a period the model has never seen.

3. OWN EVERY LINE OF CODE
   If you cannot explain it out loud, do not put it on your resume.
   Understand before you extend. Read before you write.

4. HONEST RESUME CLAIMS ONLY
   Only claim what is actually built and working.
   Per-trade P&L is not in JSON output — do not claim it is.
   Deployment means actually deployed — not running locally.

5. BUILD IN ORDER
   Do not skip phases. Each one is a foundation for the next.
   Adding agentic AI before the tools exist is just a chatbot.
   Adding ML before the data pipeline is built wastes time.

6. PYTHON IS THE PRIORITY LANGUAGE
   All ML and NLP work happens in Python.
   The FastAPI service is the most important thing to push to
   GitHub right now — even a skeleton counts.

7. THIS IS A RESEARCH TOOL, NOT A TRADING SYSTEM
   Always label it clearly on the platform.
   Never claim backtest results predict future returns.
   Paper trading via Alpaca is simulation, not real money.
