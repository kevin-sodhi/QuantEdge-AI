PHASE 5 — SIGNAL COMBINER
----------------------------------------------------------------
Goal: Merge all signals into one final trading decision.

Signal inputs:
- Price Signal (Phase 3):       -1.0 to +1.0 (weight: 40%)
- Sentiment Signal (Phase 4A):  -1.0 to +1.0 (weight: 30%)
- Event Signal (Phase 4B):      -1.0 to +1.0 (weight: 20%)
- Pattern Signal (Phase 3 CNN): -1.0 to +1.0 (weight: 10%)

Default formula:
Combined = (price x 0.4) + (sentiment x 0.3) +
           (event x 0.2) + (pattern x 0.1)

When macro event detected, weights shift:
- Event weight → 40%, Price → 35%, Sentiment → 20%, Pattern → 5%

Thresholds:
- Score above +0.3  → BUY
- Score below -0.3  → SELL
- Between           → HOLD (stay out)

Position sizing via Kelly Criterion:
- f = (bp - q) / b
- Uses ML signal confidence as win probability
- Apply half-Kelly in practice to reduce drawdown

New FastAPI endpoint:
- GET /combined-signal?ticker=AAPL

PHASE 7 — PORTFOLIO MANAGER + OPTIMIZER
----------------------------------------------------------------
Goal: The layer between signal combiner and execution engine.
Answers: given ALL tickers, how do we allocate capital?

Component 1: Signal Aggregator
- Scans 20-50 tickers simultaneously
- Runs full signal pipeline on each
- Produces ranked signal table
- Only tickers above +0.3 or below -0.3 pass through

Component 2: Capital Allocator (4 methods)
- Equal weight: split equally across BUY signals
- Signal-weighted: stronger signal = larger allocation
- Markowitz optimal: maximize Sharpe across portfolio
- Kelly Criterion: size by edge probability per position

Component 3: Correlation-Aware Allocation
- Calculate pairwise correlation across all candidates
- Above 0.85: reduce weaker signal by 60%
- 0.70-0.85: reduce weaker signal by 30%
- Below 0.70: no adjustment needed
- Prevents secretly concentrated sector bets

Component 4: Rebalancing Engine
- Trigger on 5% minimum signal change
- Require 2 consecutive days of reversal before exit
- Weekly review of all positions
- Event-triggered immediate rebalancing

Component 5: Drawdown Guard (3 tiers)
- Portfolio down 5% in 1 week → reduce all positions 30%
- Portfolio down 10% from peak → reduce 60%, no new positions
- Portfolio down 15% from peak → exit all, move to cash
- Recovery: 3 consecutive positive signal days before re-entry
- Restart at 50% normal size, scale up over 2 weeks

Component 6: Markowitz Optimizer
- scipy.optimize.minimize with SLSQP method
- Constraint: max 20% per position
- Maximize Sharpe ratio across full portfolio

Component 7: Efficient Frontier
- Plot all portfolio combinations in return/risk space
- Show current vs optimal allocation

Component 8: Monte Carlo Simulation (10,000 runs)
- Randomized parameters, starting dates, regimes
- Output: best case, expected, worst case outcomes
- p-value of strategy edge from simulation

Component 9: Multi-Strategy Comparison
- All strategies run simultaneously, own sub-portfolio each
- Compare Sharpe, drawdown, win rate, CAGR side by side

Component 10: Event-Adjusted Rebalancing
- Macro event → shift toward winners, away from losers
- Show before/after weight comparison in UI

New FastAPI endpoints:
- GET /portfolio/scan
- GET /portfolio/allocate?method=markowitz
- GET /portfolio/rebalance
- GET /portfolio/correlation-matrix
- GET /portfolio/efficient-frontier
- GET /portfolio/monte-carlo?runs=10000
- GET /portfolio/drawdown-status
- GET /portfolio/event-rebalance?event_id=123
- GET /portfolio/audit-log

================================================================
ALPHA MINING PIPELINE
================================================================
Automated system that finds, tests, validates, and deploys
new trading signals daily. This is what quant hedge funds do.

Stage 1: Alpha Generation (daily, automated)
- Claude with web search finds new research papers daily
- Extracts signal hypotheses from academic papers
- Translates into structured strategy specification JSON
- Generates Python strategy code from specification

Stage 2: Alpha Testing (automated for each candidate)
- In-sample backtest (hypothesis validation)
- Walk-forward validation (out-of-sample honesty)
- Permutation test (is the edge statistically real?)
- Cross-ticker validation (does it generalize?)
- Regime validation (does it survive market changes?)
- Transaction cost sensitivity (viable after costs?)
- Bonferroni-corrected p-value (adjust for multiple testing)

Stage 3: Alpha Diagnosis (Claude API interprets results)
- Overfitting: in-sample much better than out-of-sample
- Underfitting: both Sharpes low, no real edge
- Weak edge: real but small, may not survive costs
- Regime-dependent: works in some markets, fails others
- No edge: permutation test shows no better than random

Auto-fix loop:
- If overfitting: reduce features, add regularization,
  reduce lookback period, retest automatically
- If underfitting: add features, increase complexity, retest
- If fixable: auto-fix and retest before final decision
- WhatsApp notification at each stage

Stage 4: Alpha Deployment
- Approved signals added to platform automatically
- Appear in Strategy Lab discovery feed
- Paper trading system uses them immediately
- Track performance from day of deployment

Stage 5: Alpha Monitoring
- Live performance vs backtest prediction tracked daily
- Drawdown circuit breaker per strategy
- Signal drift detection (KS test on signal distribution)
- Regime compatibility check ongoing
- Strategy health score updated daily (0-100)

Stage 6: Alpha Decay Management
- Failing strategies deactivated automatically
- Moved to Strategy Graveyard with cause of death
- Causes: overfitting, regime change, crowding,
  transaction costs, signal drift, drawdown exceeded
- Lessons fed back to discovery criteria
- Meta-learning makes future discovery smarter

Strategy Graveyard:
- Every failed strategy logged with full post-mortem
- Date discovered, date deactivated, reason, final metrics
- Pattern analysis: what types of strategies fail most?
- This knowledge filters out bad candidates earlier

Strategy Robustness Tests (all must pass before deployment):
- Parameter sensitivity: performance stable across ±20% params
- Cost sensitivity: viable at 0.2% transaction cost minimum
- Monte Carlo permutation: beats randomly shuffled data

----------------------------------------------------------------
CRITICAL RULES — READ BEFORE EVERY SESSION
----------------------------------------------------------------

1. NO LOOKAHEAD BIAS
   Every feature must use only past data, never future.
   Most common mistake in student quant projects.
   Looks great in backtest, fails immediately in production.

2. WALK-FORWARD VALIDATION IS MANDATORY
   Never evaluate a strategy on data it trained on.
   In-sample results are always optimistic.
   Out-of-sample results are what you actually get.

3. OWN EVERY LINE OF CODE
   Cannot explain it out loud = do not put it on resume.
   Understand before you extend. Read before you write.

4. HONEST RESUME CLAIMS ONLY
   Only claim what is actually built and working.
   Paper trading means Alpaca paper, not real money.
   Deployed means live URL, not running on localhost.

5. BUILD IN ORDER
   Do not skip phases. Each is the foundation for the next.
   Paper trading before agents are built = no intelligence.
   Agents before tools are built = just a chatbot.

6. PYTHON IS THE PRIORITY LANGUAGE
   All ML, NLP, agents, and paper trading in Python.
   Push FastAPI skeleton to GitHub now — even 50 lines.

7. PAPER TRADING IS PROOF, NOT PRODUCT
   This is a research and educational platform.
   Paper trading proves the algorithms work statistically.
   It is not investment advice. Label everything clearly.

8. EVENT ANALYSIS EDGE IS NOT SPEED
   You cannot beat institutional traders on reaction time.
   Your edge is second-order analysis and position sizing.
   Never claim otherwise on the platform or in interviews.

9. STATISTICAL SIGNIFICANCE IS NON-NEGOTIABLE
   A good-looking backtest may be luck.
   Permutation test + walk-forward + Bonferroni correction.
   Confirm edge is real before deploying or claiming it.

10. START THE TRACK RECORD IMMEDIATELY
    Create Alpaca paper account today.
    Even running basic momentum strategy from day 1 counts.
    Every day of documented history = stronger proof.
    By August 2026 you want months of real data, not days.

11. THE GRAVEYARD IS AS IMPORTANT AS THE WINNERS
    Log every failed strategy with cause of death.
    Showing you understand why strategies fail is as
    impressive as showing strategies that work.
    Honest analysis of failure = intellectual maturity.

12. AGENTS MUST USE TOOLS, NOT MEMORY
    Claude API system prompt must force tool usage.
    Agent should never answer from general knowledge.
    Every claim must be backed by a real tool call result.
    This is what makes agent outputs trustworthy.
